Compared with Redis

Time complexity

Some functions have different time complexity, check the orders here.

Behaviour

  • SAVE, BGSAVE, LASTSAVE
    This commands are database dependent. The original Redis saves all databases at once, edis saves just the one you SELECT'ed
  • INFO
    Provides much less information and no statistics (so, CONFIG RESETSTAT does nothing at all)
  • MULTI doesn't support
    - cross-db commands (i.e. FLUSHALL, SELECT, MOVE)
    - non-db commands (i.e AUTH, CONFIG *, SHUTDOWN, MONITOR)
    - pub/sub commands (i.e. PUBLISH, SUBSCRIBE, UNSUBSCRIBE, PSUBSCRIBE, PUNSUBSCRIBE)
  • (P)UNSUBSCRIBE
    This commands are not allowed outside PUBSUB mode
  • PUBLISH
    Its Response is not precise: It's the amount of all clients subscribed to any channel and/or pattern, not just those that will handle the message. On the other hand it runs in O(1) because it's asynchronous, it just dispatches the message.
  • SPOP
    This command always returns an specific element of the set instead of a random one. That was needed to make the function really O(1).

Missing Features

  • Dynamic node configuration (i.e. the SLAVEOF command is not implemented)
  • Encoding optimization (i.e. all objects are encoded as binary representations of erlang terms, so for instance "123" will never be stored as an int)
  • OBJECT REFCOUNT allways returns 1 for existing keys and (nil) otherwise

Unsupported Commands

SYNC, SLOWLOG, SLAVEOF, DEBUG *
